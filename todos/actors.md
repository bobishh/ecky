# Семья Эйнаксов — мульти-агентная архитектура

## Концепция

Три персонажа с разными ролями, моделями и ценой. Общая база (SQLite), общий MCP endpoint (Эки хост). Бэки и третий дергают Эки через MCP, Эки оркестрирует FreeCAD и LLM.

---

## Акторы

### 🟠 Эки (Ecky) — главный дизайнер
- **Статус:** существует
- **Модель:** дорогая (Gemini Pro / Claude Opus / GPT-4)
- **UI:** основное Tauri-приложение, мужской/нейтральный образ, VertexGenie
- **Роль:** творческая генерация, крупные редизайны, форки, семантический манифест
- **Триггеры:** новый промпт пользователя, fork, import
- **Задачи:**
  - Первичная генерация macro + uiSpec + семантика
  - Большие переделки ("сделай квадратным", "добавь крышку")
  - Решение о маршруте → сам делает или делегирует третьему

### 🔵 Бэки (Becky) — сестра, внешний координатор
- **Статус:** не существует, новый продукт
- **Модель:** та же дорогая что у Эки (или Claude-specific если в IDE)
- **UI:** отдельное Tauri-приложение с женским образом, та же trait-генетика что у Эки
- **Роль:** мост между сторонними агентами (Claude Code, Cursor, IDE) и Эки
- **Подключение:** MCP клиент → `http://127.0.0.1:39249/mcp`
- **Ключевая фича:** `ask_user` петля — вместо вопроса в чате → кнопка в Бэки UI
- **Задачи:**
  - Принимать команды от IDE-агентов и роутить в Эки
  - Показывать confirmation dialogs когда агент просит подтверждения
  - Отображать прогресс и историю сессий от имени агента
  - Визуальный "посредник" между человеком и внешним агентом

## Петля ask_user через MCP

Когда внешний агент (например Claude Code) хочет подтверждения от пользователя — вместо вопроса в терминале он дергает MCP tool `user_confirm_request`. Бэки (или Эки) показывает UI-кнопки.

```
Внешний агент (Claude Code)
  → MCP tool: user_confirm_request({ message, buttons: ["Да", "Нет", "Пропустить"] })
  → Rust: emit Tauri event "agent-confirm-request" → Frontend
  → Frontend: Modal с кнопками (не блокирует основной UI)
  → Пользователь жмёт кнопку
  → Tauri command: resolve_agent_confirm(request_id, choice)
  → Rust: oneshot channel → MCP handler получает ответ
  → MCP: возвращает { "choice": "Да" } агенту
  → Агент продолжает работу
```

### Что нужно для реализации

**Backend (Rust):**
- [ ] Добавить `AppHandle` в `HttpServerState` (сейчас только `AppState` + `PathResolver`)
- [ ] Добавить `confirm_channels: Arc<Mutex<HashMap<String, oneshot::Sender<String>>>>` в `AppState`
- [ ] Новый MCP tool: `user_confirm_request({ requestId, message, buttons, timeoutSecs? })`
- [ ] Новый Tauri command: `resolve_agent_confirm(request_id, choice)` → резолвит канал
- [ ] Таймаут: если пользователь не ответил за N секунд → дефолтный ответ или ошибка

**Frontend (Svelte):**
- [ ] Слушать Tauri event `agent-confirm-request` в `App.svelte`
- [ ] Показывать confirmation panel (не modal — нотификация с кнопками, чтобы не блокировать UI)
- [ ] Вызывать `resolve_agent_confirm` при нажатии
- [ ] Стек из нескольких pending confirmations (если агент спрашивает несколько раз)

## Роутинг запросов (intent-based dispatch)

```
Пользователь → prompts
                ├── creative / structural   → Эки (дорогая модель)
                ├── parametric / minor fix  → Третий (дешевая модель)
                └── question / explain      → Эки (light model уже есть)

Внешний агент → MCP
                ├── macro_replace_and_render  → Эки
                ├── params_patch_and_render   → Третий (если не задан явно)
                └── user_confirm_request      → Бэки UI / Эки UI
```

Классификатор intent уже есть в `requestOrchestrator.ts` — нужно добавить третий класс `parametric-only` и маппинг на дешевую модель.

Рекомендация: **начать с Варианта А** — быстрее, меньше дублирования кода, основная ценность в петле `ask_user` и красивом женском образе.

## Модельная матрица (Эки как AGI-оркестратор)

Эки имеет доступ ко всем провайдерам (Google/OpenAI/Ollama/любой OpenAI-compatible). Это позволяет
строить **agent MoE поверх API** — подменять модели по задаче, не меняя архитектуру.
Большие лабы делают то же самое на уровне весов (Gemini Ultra = ensemble, GPT-4 = MoE).
Эки делает это на уровне роутинга — что даёт гибкость и независимость от одного провайдера.

### Матрица по ролям

| Задача | Модель | Почему |
|--------|--------|--------|
| Первичный дизайн, крупный редизайн | Gemini Pro / Claude Opus | нужна глубина, пространственное мышление |
| Параметрические правки, Тики | Gemini Flash / Claude Haiku | скорость, цена, задача простая |
| Intent classification | light_model (уже есть в конфиге) | 2 класса, минимальный контекст |
| Вопросы о дизайне (explain/why) | light_model | только текст, нет генерации |
| Визуальный черновик (image gen) | Imagen 3 / DALL-E 3 / Flux | превью до FreeCAD рендера |
| Сложная геометрия / reasoning | o3 / Gemini Thinking | нужен chain-of-thought |
| Локальная модель (privacy режим) | Ollama (llama3, qwen2.5-coder) | уже поддерживается |

### Идея: быстрый визуальный черновик

Сейчас: промпт → FreeCAD макро → рендер (~10-30 сек) → превью

С image gen: промпт → image gen (~2 сек) → показать пользователю → "нравится?" → тогда FreeCAD

Это снижает стоимость итерации на ранних этапах в 10x. Пользователь не ждёт FreeCAD
пока не определился с концепцией.

**Реализация:** новый intent класс `sketch` → вызов image gen API → отображение в viewer
как "concept preview" с плашкой "Not a 3D model" → кнопка "Generate 3D from this".

### Идея: thinking модель для сложных задач

Некоторые геометрические задачи требуют reasoning:
- "Сделай так чтобы крышка защёлкивалась"
- "Добавь резьбу M8"
- "Оптимизируй под минимальный вес при сохранении прочности"

Для таких задач: автоматически переключаться на thinking модель (o3/Gemini Thinking)
когда intent classifier определяет высокую геометрическую сложность.

Флаг в конфиге: `thinking_model` — опциональный, только для сложных задач.
